Self-Improving Agents / Project Proposal

项目导语压缩版：先做仿真闭环，Robot4Robot 后移

这里替代的是 Self-Improving Agents 项目自己的长 intro：proposal 已经展开，导语只保留方向判断和一两句话。

Proposal：把 Self-Improving Agents 做成 AgenticSim 的 simulation-first 系统。Scheduler 选择任务和难度，Scene Generator 生成环境和扰动，Train Agent 训练策略或 workflow，Evaluate Agent 用 verifier 评分，Analyze Agent 把失败变成 memory，再反向更新 task schedule。真机和 Robot4Robot 是后续 promotion layer。

为什么不先真机

真实机器人调试、采集、安全 gate 和失败复现成本都高。把 analyze、train、evaluate、schedule task 都压到真机上，会让第一阶段被工程复杂度拖住。

为什么先仿真

仿真可以并行生成 scene、固定 seed、快速评估、复现实验，并把失败归因转化为可持续变难的 benchmark 和 curriculum。

本周可讲的 demo

先做小闭环：任务族 -> 场景变体 -> rollout -> verifier -> failure memory -> 下一轮 schedule。

Reference reading

调研结论：关键不是生成数据，而是生成可执行环境

General Agent 和 EvoEnv 都支持这个 proposal 方向：把 self-improvement 从“多一点 synthetic data”改成“持续构造能训练模型的 verifier environment”。

General Agent：synthesizer / solver 循环

Prime Intellect 的 General Agent 用 Synthesizer 生成任务族和难度 tier，用 Solver 估计 pass rate，只有落在目标难度区间的任务才进入 corpus。

迁移到机器人：每个任务也应该有 scene state、controller/tool API、instruction、reference rollout、verifier 和 difficulty tier。

EvoEnv：environment-construction loop

EvoEnv 强调每个 artifact 应是 executable object：能 sample instance、compute reference、score response，并经过 staged validation、semantic review、difficulty calibration 和 novelty checks。

迁移到机器人：生成场景必须能重放、能评分、能产生可诊断失败；只视觉上合理不够。

DB / tools / instruction / verification Scene state / robot controllers / task goal / executable simulator verifier AgenticSim 任务必须是可执行对象，不是 prose-only benchmark。
Pass-rate calibrated difficulty 用 baseline policy 或 VLA solver 的 rollout success rate 标定难度 自动接受“刚好难”的任务，拒绝太简单、坏掉或不可复现的任务。
Stable solve-verify asymmetry Verifier 容易执行，但策略在新场景里仍然容易失败 reward 不会立刻失效，环境池可以持续提供训练信号。

系统图：AgenticSim simulation-first self-improvement loop

闭环架构图

agent actions learning feedback later real-world promotion
1 Schedule task

选择任务族、难度、seed、目标 failure mode 和 evaluator budget。

2 Generate scene

生成 object layout、材质、遮挡、扰动和物理参数，保留 manifest。

3 Train

训练策略、微调 workflow、更新 prompt/tool 调用或 recovery policy。

4 Evaluate

跑 rollout、verifier、safety check、difficulty gate 和 regression suite。

5 Analyze failure

归因到 grasp、collision、contact、planning、perception 或 instruction mismatch。

6 Update memory

写入 failure memory、task curriculum、accepted/rejected scenes，触发下一轮。

Phase 0 整理环境底座

固化 129 个环境，统一 task schema、seed、asset version、verifier 和 rollout command。

Phase 1 Scene generation

从任务族和 failure type 生成场景变体，先覆盖 tabletop / household manipulation。

Phase 2 Evaluate + analyze

跑 rollout 后自动归因失败，记录 trace memory，用 verifier 决定 accept、reject 或 mutate。

Phase 3 Training signal

用 accepted scene 和 failure memory 做 SFT/RL 或 workflow tuning，比较新旧 agent。

Phase 4 Real promotion

只把通过仿真 gate 的少量任务交给真实机器人验证。Robot4Robot 在这里出现。
